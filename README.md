# Energy Optimization Project

An end-to-end MLOps MVP for industrial energy scheduling.

This project demonstrates how hourly electricity prices, weather signals, production load, energy forecasting, and optimization logic can be combined into a decision support application. The system recommends the lowest cost four hour production window for an industrial process and compares that recommendation with a standard baseline schedule.

The core pipeline is:

```
data -> features -> forecast -> optimization -> API -> dashboard -> Docker
```

## Project Purpose

Industrial electricity prices can vary significantly throughout the day. For companies running energy-intensive processes, even small scheduling changes can reduce operating costs without changing the underlying production process.

The goal of this MVP is to answer one focused question:

> If a process must run every day for four consecutive hours, when should it run to minimize estimated energy cost?

The application currently supports:

- Loading simulated hourly industrial energy data
- Training and loading an energy usage forecasting model
- Predicting hourly energy consumption
- Finding the cheapest valid production window per day
- Comparing optimized schedules against a baseline `08:00-12:00` run
- Serving the workflow through a FastAPI backend
- Presenting the recommendation in a Streamlit dashboard
- Running the API and dashboard with Docker Compose

## MVP Goals

This is intentionally scoped as a clear MVP rather than a general energy optimization platform.

The MVP focuses on:

- A fixed industrial process duration, currently four hours
- Hourly electricity prices in `SEK/kWh`
- Simulated production and weather data
- Forecasted hourly energy usage in `kWh`
- Daily schedule optimization
- A dashboard suitable for demo and portfolio presentation

Future versions could connect to live spot-price APIs, real factory data, shift calendars, machine constraints, and production planning systems.

## Generated Data

The project uses simulated data stored in:

```text
data/simulated/hourly_energy_data.csv
```

The included dataset contains 2,160 hourly rows, equal to 90 days of data.

Columns:

| Column | Meaning |
| --- | --- |
| `timestamp` | Hourly timestamp |
| `electricity_price` | Simulated electricity price in `SEK/kWh` |
| `temperature` | Simulated outdoor temperature |
| `production_load` | Production level: `low`, `medium`, or `high` |
| `historical_energy_usage` | Historical process energy usage in `kWh` |

The simulation is designed to be realistic enough for MVP development:

- Prices are higher during morning and evening peaks
- Prices are lower during night and midday periods
- Temperature follows seasonal and daily variation
- Production load affects energy usage

The data is generated by:

```text
src/data_generator.py
```

This MVP uses simulated industrial energy and price data to demonstrate the optimization workflow. It should not be interpreted as real operational or market data.

## Core Workflow

1. `src/data_loader.py` loads and validates the simulated hourly CSV.
2. `src/features.py` creates ML-ready features from timestamps and production load.
3. `src/forecasting.py` trains or loads a model and predicts `predicted_energy_usage_kwh`.
4. `src/optimizer.py` calculates hourly cost and finds the cheapest valid four-hour window per day.
5. `src/api.py` exposes the data, forecast, optimization, and recommendation endpoints.
6. `dashboard/app.py` visualizes the daily forecast, recommendation, savings, and alternative windows.

## Forecasting

The forecasting module supports:

- Linear Regression
- Random Forest Regressor
- MAE and RMSE evaluation
- Model saving and loading through `joblib`

The saved model is stored at:

```text
models/energy_model.pkl
```

Train or refresh the model with:

```bash
uv run python scripts/train_model.py
```

## Optimization Logic

The optimizer is built around the assumption that the process must run every day.

For each day, it:

1. Sorts the hourly data by timestamp
2. Tests every valid consecutive four-hour window
3. Calculates total estimated cost
4. Selects the cheapest window
5. Compares it with the baseline `08:00-12:00` schedule
6. Returns estimated savings in SEK and percent

Cost calculation uses this priority:

1. `predicted_energy_usage_kwh` if forecast data is available
2. `historical_energy_usage` if forecast data is not available
3. A constant fallback energy usage from `src/config.py`

## API

The FastAPI app is defined in:

```text
src/api.py
```

Available endpoints:

| Method | Endpoint | Description |
| --- | --- | --- |
| `GET` | `/` | Service overview |
| `GET` | `/health` | Health check and configured paths |
| `GET` | `/data` | Latest rows from the loaded dataset |
| `GET` | `/forecast` | Forecasted energy usage for a selected date |
| `POST` | `/optimize` | Optimized schedule based on request parameters |
| `GET` | `/recommendation` | Default optimization recommendation |

Example forecast request:

```bash
curl "http://localhost:8000/forecast?date=2026-03-31&limit=24"
```

Example optimization request:

```bash
curl -X POST "http://localhost:8000/optimize" \
  -H "Content-Type: application/json" \
  -d '{
    "process_duration_hours": 4,
    "earliest_start": 6,
    "latest_end": 22,
    "baseline_start": 8,
    "baseline_end": 12
  }'
```

## Running Locally

Install dependencies with `uv`:

```bash
uv sync
```

Train the model:

```bash
uv run python scripts/train_model.py
```

Run the API:

```bash
uv run uvicorn src.api:app --reload 
```

Run the dashboard:

```bash
uv run streamlit run dashboard/app.py
```

Local URLs:

```text
API:       http://localhost:8000
API docs:  http://localhost:8000/docs
Dashboard: http://localhost:8501
```

## Running With Docker

Build and start both services:

docker compose up --build -d

| Service | URL |

## Current Limitations

- The dataset is simulated and not connected to live electricity markets.
- The forecast currently predicts on the available hourly data rather than a true future weather/production plan.
- The optimizer assumes one process with a fixed duration.
- Operational constraints such as maintenance windows, staffing, machine warmup, and production deadlines are not yet modeled.
- No database is used; CSV storage is enough for this MVP.
